# v0.4.0 вАФ 4K permission gating + localized service descriptions

**Status:** design approved 2026-05-06
**Branch:** `v0.4-4k-i18n` (local only)
**Target release:** `v0.4.0`

## Goals

1. **4K permission gating** вАФ reject 4K requests at the HA boundary when the mapped Seerr user lacks the required permission, instead of letting Seerr return a generic API error.
2. **Localized service descriptions** вАФ full EN+BG parity for every service field shown in the HA frontend, including the new 4K rejection error message.
3. **BG voice intent updates** вАФ replace the bookish verb `–њ–Њ–Є—Б–Ї–∞–є` with the natural Bulgarian `—Б–≤–∞–ї–Є` ("download") and `–љ–∞–Љ–µ—А–Є` ("find") across all `RequestMedia` patterns. This also resolves the grammar bug in the existing `(–Љ–Њ–ґ–µ—И –ї–Є –і–∞|–Љ–Њ–ї—П) –њ–Њ–Є—Б–Ї–∞–є` pattern.
4. **Card 4K toggle** вАФ global card-level toggle so users can request 4K from the Lovelace card, not just via Developer Tools.

All four ship in the same release: 1 enables backend rejection, 2 makes the rejection user-readable, 3 makes the BG voice path natural, and 4 surfaces 4K to the primary UI users actually use.

## Non-goals

- Permission-aware UI in the card (hide/disable the 4K toggle for users without 4K perms) вАФ defer to v0.4.1. The server-side rejection with localized error already covers the bad-click case; smart UI is polish.
- Caching of the permission lookup вАФ YAGNI; 4K requests are infrequent, an extra GET is ~50ms. Add later if anyone reports latency.
- Card string localization вАФ card UI strings (`Request`, `SearchingвА¶`) are hardcoded English today and stay that way. Localizing the card is its own project; the new `4K` label is identical in EN and BG.
- Locales beyond EN+BG вАФ out of scope; matches the existing project pattern (intents/bg.yaml + intents/en.yaml only).
- Multi-instance support вАФ separate v0.4 candidate, deferred.

## 4K permission gating

### Behaviour

When a service call to `haseerr.request` includes `is_4k=true`:

1. Resolve the caller's Seerr user via existing `_resolve_user_id()` flow.
2. Fetch that user's permission bitmask via `GET /api/v1/user/<id>` (new SeerrClient method).
3. Decide:
   - `permissions & 2` (ADMIN) вЖТ allow, regardless of media type.
   - `permissions & 4096` (REQUEST_4K, global) вЖТ allow.
   - `media_type=movie` and `permissions & 8192` (REQUEST_4K_MOVIE) вЖТ allow.
   - `media_type=tv` and `permissions & 16384` (REQUEST_4K_TV) вЖТ allow.
   - `media_type=music` вЖТ reject (Seerr/Lidarr has no 4K concept; sanity guard).
   - Otherwise вЖТ reject.
4. On reject: raise `HomeAssistantError` with `translation_domain="haseerr"`, `translation_key="not_authorized_4k"`, and `translation_placeholders={"media_type": media_type}`. HA renders the localized message from `translations/<locale>.json::exceptions.not_authorized_4k.message`.

When `is_4k` is false (the common path): no permission lookup is performed. Zero overhead.

### Code structure

- **`hub.py`** (~15 LOC):
  - New method `SeerrClient.get_user_permissions(user_id: int) -> int` returning the `permissions` field from `/api/v1/user/<id>`.
  - New exception `SeerrPermissionError(SeerrError)`.

- **`const.py`** (~4 LOC):
  - `PERM_ADMIN = 2`
  - `PERM_REQUEST_4K = 4096`
  - `PERM_REQUEST_4K_MOVIE = 8192`
  - `PERM_REQUEST_4K_TV = 16384`

- **`services_impl.py`** (~25 LOC):
  - New helper `async def _check_4k_permission(client, user_id, media_type)`. Raises `SeerrPermissionError` on denial.
  - In `_request()`: if `call.data.get("is_4k")`, call `_check_4k_permission` after `_resolve_user_id` and before `client.request()`.
  - Catch `SeerrPermissionError` and re-raise as `HomeAssistantError` with translation key + placeholder.

### Tests (`tests/test_hub.py`, `tests/test_services.py`)

- `test_get_user_permissions` вАФ fixture user, parses bitmask correctly.
- `test_request_4k_movie_unauthorized` вАФ `permissions=32` (REQUEST only), `is_4k=true`, `media_type=movie` вЖТ `HomeAssistantError` raised, **no** POST to `/api/v1/request` made (verified via mock call count).
- `test_request_4k_movie_authorized_global` вАФ `permissions=4096|32` вЖТ submits successfully, body contains `is4k: true`.
- `test_request_4k_movie_authorized_per_type` вАФ `permissions=8192|32` вЖТ submits successfully.
- `test_request_4k_tv_authorized_per_type` вАФ `permissions=16384|32` вЖТ submits successfully.
- `test_request_4k_movie_admin_bypass` вАФ `permissions=2` вЖТ submits successfully without checking 4K bits.
- `test_request_4k_music_rejected` вАФ `is_4k=true`, `media_type=music` вЖТ rejected, no permission GET, no POST.
- `test_request_non_4k_skips_permission_check` вАФ `is_4k=false` вЖТ **zero** GETs to `/api/v1/user/<id>` (asserts no overhead on the common path).

## Localized service descriptions

### Coverage

`services.yaml` stays as-is (English source-of-truth; used by HA as fallback if translations fail to load).

`translations/en.json` вАФ additions to the existing `services` block:
- `services.request.fields.is_4k` вАФ name + description
- `services.request.fields.title` вАФ name + description
- Full `services.user_quota` block вАФ service name + description + `user_override` field

`translations/bg.json` вАФ currently has no `services` block. Add the full block in Bulgarian, mirroring en.json structure. Covers all 5 services (`search`, `request`, `approve_request`, `decline_request`, `user_quota`) and every field defined in services.yaml.

Both files вАФ new top-level `exceptions` block:

```jsonc
// en.json
"exceptions": {
  "not_authorized_4k": {
    "message": "Not authorized to request 4K {media_type}."
  }
}
```

```jsonc
// bg.json
"exceptions": {
  "not_authorized_4k": {
    "message": "–Э—П–Љ–∞—В–µ —А–∞–Ј—А–µ—И–µ–љ–Є–µ –Ј–∞ –Ј–∞—П–≤–Ї–∞ –љ–∞ 4K {media_type}."
  }
}
```

### Translation principles

- BG tone matches existing `bg.json` (formal, passive where appropriate).
- Reuse terminology from `intents/bg.yaml` for consistency: "—Д–Є–ї–Љ" (movie), "—Б–µ—А–Є–∞–ї" (TV show), "–Ј–∞—П–≤–Ї–∞" (request), "–њ–Њ—В—А–µ–±–Є—В–µ–ї" (user).
- The `{media_type}` placeholder stays English (`movie` / `tv`) in both locales вАФ these values are part of the service-call API and not user-facing concepts in the way the rest of the strings are. Localizing them would require a parallel translation step that adds complexity for no UX gain.

## BG voice intent updates

`custom_components/haseerr/intents/bg.yaml` вАФ `RequestMedia.data[0].sentences` becomes:

```yaml
- "—Б–≤–∞–ї–Є {title}"
- "–љ–∞–Љ–µ—А–Є {title}"
- "–і–Њ–±–∞–≤–Є {title} –≤ (Seerr|Plex|Jellyfin)"
- "–Љ–Њ–ґ–µ—И –ї–Є –і–∞ (—Б–≤–∞–ї–Є—И|–љ–∞–Љ–µ—А–Є—И) {title}"
- "–Љ–Њ–ї—П (—Б–≤–∞–ї–Є|–љ–∞–Љ–µ—А–Є) {title}"
- "(—Б–≤–∞–ї–Є|–љ–∞–Љ–µ—А–Є) {title} —Б–µ–Ј–Њ–љ {season}"
```

Removed entirely:
- `"–њ–Њ–Є—Б–Ї–∞–є {title}"` вАФ bookish/formal, replaced by natural alternatives
- `"(–Љ–Њ–ґ–µ—И –ї–Є –і–∞|–Љ–Њ–ї—П) –њ–Њ–Є—Б–Ї–∞–є {title}"` вАФ grammar bug + bookish verb, replaced
- `"–њ–Њ–Є—Б–Ї–∞–є {title} —Б–µ–Ј–Њ–љ {season}"` вАФ replaced with alternation form

The `–і–Њ–±–∞–≤–Є {title} –≤ (Seerr|Plex|Jellyfin)` pattern stays untouched (uses a different verb that's already natural).

The grammar bug is implicitly fixed: alternations like `(—Б–≤–∞–ї–Є—И|–љ–∞–Љ–µ—А–Є—И)` after `–і–∞` correctly use the present-tense form.

Out of scope (deferred):
- Cyrillicized brand-name fallbacks for `(Seerr|Plex|Jellyfin)` in the `–і–Њ–±–∞–≤–Є` pattern
- Additional confirm/cancel variants

## Card 4K toggle

`custom_components/haseerr/www/haseerr-card.js`:

**Visual change** вАФ add a small toggle row above the search input:

```
[ ] 4K                     вЖР new toggle, off by default
[ Search boxвА¶    ] [рЯФН]
```

When the toggle is on, every `Request` button click submits with `is_4k: true`.

**Implementation:**
- New reactive state property `_is4k = false`.
- New element in `render()` above the existing `.input-row`: a `<label>` with `<input type="checkbox">` bound to `_is4k`.
- `_onPick(result, seasons)` includes `is_4k: this._is4k` in the service-call data when the toggle is on.
- Success toast suffix: ` ¬Ј 4K` when `_is4k` is true (e.g. `Requested Inception for Alice вАФ pending ¬Ј 4K`).
- Music results never submit `is_4k=true` вАФ guard inside `_onPick` (`if (result.media_type !== "music") data.is_4k = this._is4k`). Belt-and-suspenders with the server-side guard.

**Error handling:**
- The existing `try/catch` around `callService` catches `HomeAssistantError` (including the new localized 4K rejection). HA includes the localized message in `e.message`; the existing `Request failed: ${e.message}` toast renders it correctly.

**Editor (`getConfigElement`/GUI editor):** no change. `is_4k` is a per-session UI state, not a config option to persist.

### Tests

The card has no automated tests today (Lit component, no JS test infra in the project). Manual verification only вАФ covered by the existing smoke-test workflow on lk HA VM 900.

### Tests (new `tests/test_translations.py`)

- `test_en_translations_cover_all_services` вАФ every service in `services.yaml` has a matching entry in `en.json::services` with `name`, `description`, and one entry per field.
- `test_bg_structurally_matches_en` вАФ every key path under `services` in `en.json` exists at the same path in `bg.json` (value comparison not done; just key parity).
- `test_exceptions_block_present` вАФ `exceptions.not_authorized_4k.message` exists in both locales and contains the `{media_type}` placeholder.

These tests catch drift on future additions: anyone adding a new service field will see CI fail until they add translations.
